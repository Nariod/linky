PHILOSOPHIE
───────────
Ce roadmap suit deux principes :
  • KISS — chaque feature doit être la solution la plus simple qui fonctionne.
  • MVP  — livrer un C2 utilisable en engagement réel, pas un clone de Mythic.

Linky ne vise PAS la parité avec Sliver/Mythic/Havoc. L'avantage compétitif est :
  1. Rust natif (pas de Go/Python runtime, RE plus difficile)
  2. Codebase lisible et auditable (~2k LOC vs ~50k pour Sliver)
  3. Déploiement conteneurisé en 3 commandes

Priorités : P0 = bloquant sécurité, P1 = MVP, P2 = nice-to-have, P3 = futur.

[🟡] 1.6.1  CARGO.TOML DES IMPLANTS — DÉPENDANCES ORPHELINES
         Depuis la factorisation 1.5.1, les Cargo.toml de linux/windows/osx
         déclarent des dépendances dont le code est maintenant dans link-common :
           - aes-gcm  : plus utilisé directement dans stdlib.rs
           - sha2     : idem
           - hex      : idem
           - rand     : idem (sauf windows pour random dans inject)
           - chrono   : idem
           - serde    + serde_json : idem (sauf windows pour base64 inject)
         Le compilateur peut les garder silencieusement via link-common,
         mais elles gonflent inutilement le temps de compilation.
         FIX : retirer les dépendances non utilisées directement dans stdlib.rs.
         Vérifier avec `cargo +nightly udeps -p link-linux` etc.
         Fichiers : links/linux/Cargo.toml, links/osx/Cargo.toml
                    (links/windows/Cargo.toml garde base64 pour inject_cmd)
         Difficulté : triviale (~10 min)
