PHILOSOPHIE
───────────
Ce roadmap suit deux principes :
  • KISS — chaque feature doit être la solution la plus simple qui fonctionne.
  • MVP  — livrer un C2 utilisable en engagement réel, pas un clone de Mythic.

Linky ne vise PAS la parité avec Sliver/Mythic/Havoc. L'avantage compétitif est :
  1. Rust natif (pas de Go/Python runtime, RE plus difficile)
  2. Codebase lisible et auditable (~3k LOC vs ~50k pour Sliver)
  3. Déploiement conteneurisé en 3 commandes

Priorités : P0 = bloquant sécurité, P1 = MVP, P2 = nice-to-have, P3 = futur.

[🔴] 1.0.5  BUG CRITIQUE : DÉSÉRIALISATION IMPOSSIBLE DU PAYLOAD CHIFFRÉ
         ─────────────────────────────────────────────────────────────────
         C'EST LE BUG QUI CASSE TOUT LE C2 ACTUELLEMENT.

         Cause racine :
         Le serveur (routes.rs stage3_handler, Lock 3) chiffre un struct
         ResponsePayload qui contient {q, tasking, file, filename, upload,
         upload_path} — SANS le champ x_request_id.

         L'implant (linux/osx/windows stdlib.rs) déchiffre le payload et
         tente de le parser comme TaskResponse :

           serde_json::from_str::<TaskResponse>(&decrypted_json)

         Or TaskResponse dans link-common/lib.rs a :

           pub x_request_id: String,   // PAS de #[serde(default)] !

         Serde exige ce champ dans le JSON. Comme il est absent du payload
         chiffré, la désérialisation échoue SYSTÉMATIQUEMENT avec :
         "missing field `x_request_id`"

         Le code tombe dans le fallback :
           decrypted_task = TaskResponse {
               q: String::new(),        // ← VIDE
               tasking: String::new(),
               ...
           };

         Conséquence : decrypted_task.q est toujours vide → l'implant
         ne voit jamais de commande → boucle idle indéfiniment.
         Côté serveur, les tâches passent InProgress mais ne reçoivent
         jamais de résultat → restent InProgress → jamais affichées.

         FIX (KISS, 2 options au choix) :
         Option A (rapide, ~2 min) :
           Ajouter #[serde(default)] sur x_request_id dans TaskResponse
           (links/common/src/lib.rs). Le x_request_id du payload interne
           sera "" mais n'est pas utilisé — l'implant utilise celui de
           l'enveloppe HTTP externe (task.x_request_id).

         Option B (propre, ~15 min) :
           Créer un struct DecryptedPayload dans link-common qui correspond
           exactement à ResponsePayload du serveur (sans x_request_id,
           sans data). Modifier les 3 implants pour parser comme
           DecryptedPayload au lieu de TaskResponse.

         Recommandation : Option B est plus propre mais Option A suffit
         pour le MVP. Faire A maintenant, B dans le refactor 1.5.1.

         Fichiers : links/common/src/lib.rs (TaskResponse),
                    OU links/*/src/stdlib.rs (parsing du payload)
         Difficulté : triviale (Option A) ou faible (Option B)
         Impact : BLOQUANT — sans ce fix, aucune commande n'est exécutée.

[🟡] 1.0.6  BUG UX : show_completed_task_results RÉ-AFFICHE EN BOUCLE
         Dans cli.rs, show_completed_task_results() est appelée à chaque
         itération de la boucle interact(), avant readline(). Elle parcourt
         TOUTES les tâches Completed du link sans tracker celles déjà
         affichées. Résultat : si une tâche est Completed, elle est
         ré-affichée à chaque frappe clavier.
         FIX : Ajouter un champ `displayed: bool` dans Task (tasks.rs),
         le mettre à true après affichage, et ne montrer que les tâches
         non-displayed. Alternativement, utiliser un index "last displayed"
         dans la boucle interact().
         Fichiers : server/src/cli.rs (show_completed_task_results),
                    server/src/tasks.rs (ajouter champ displayed)
         Difficulté : faible (~15 min)
