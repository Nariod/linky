PHILOSOPHIE
───────────
Ce roadmap suit deux principes :
  • KISS — chaque feature doit être la solution la plus simple qui fonctionne.
  • MVP  — livrer un C2 utilisable en engagement réel, pas un clone de Mythic.

Linky ne vise PAS la parité avec Sliver/Mythic/Havoc. L'avantage compétitif est :
  1. Rust natif (pas de Go/Python runtime, RE plus difficile)
  2. Codebase lisible et auditable (~3k LOC vs ~50k pour Sliver)
  3. Déploiement conteneurisé en 3 commandes

Priorités : P0 = bloquant sécurité, P1 = MVP, P2 = nice-to-have, P3 = futur.

── BUGS CRITIQUES DÉCOUVERTS PAR AUDIT (P0, BLOQUANTS) ───────────────────────

[✅] 1.0.1  BUG CRITIQUE : DÉRIVATION DE CLÉ INCOMPATIBLE SERVEUR ↔ IMPLANTS
         Le serveur (routes.rs) utilise HKDF-SHA256 pour derive_key().
         Les implants (link-common/lib.rs) utilisent SHA-256 brut.
         → Les clés dérivées sont DIFFÉRENTES.
         → Le chiffrement payload (item 1.2) est SILENCIEUSEMENT CASSÉ.
         → Les commandes ne sont jamais déchiffrées par l'implant.
         → Les résultats ne sont jamais déchiffrés par le serveur.
         Le système semble fonctionner car les fallbacks sont gracieux
         (decrypted_q = "" → pas de commande → idle loop).
         FIX : Aligner server/src/routes.rs::derive_key() sur
         link-common::derive_key() (SHA-256 simple) OU inversement migrer
         link-common vers HKDF. La solution la plus simple (KISS) est
         d'aligner le serveur sur l'implant.
         Fichiers : server/src/routes.rs (derive_key, encrypt_payload,
                    decrypt_payload)
         Difficulté : faible (~30 min) mais CRITIQUE — tout le sprint 1
         dépend de ça.

[✅] 1.0.2  BUG : CALLBACK JAMAIS CHIFFRÉ À LA GÉNÉRATION
         generate.rs passe CALLBACK en clair via env var cargo.
         L'implant tente decrypt_config(CALLBACK, &key) qui échoue
         et tombe en fallback sur le plaintext.
         → Le callback IP:port est en clair dans le binaire.
         FIX : generate.rs doit chiffrer le CALLBACK avec la clé dérivée
         du IMPLANT_SECRET avant de le passer comme env var.
         Fichiers : server/src/generate.rs, links/*/build.rs (rien à
                    changer côté build.rs, juste generate.rs)
         Difficulté : faible (~30 min)
         Dépend de 1.0.1 (la dérivation de clé doit fonctionner).

[✅] 1.0.3  BUG : ZEROIZE INEFFICACE PARTOUT DANS LINK-COMMON
         3 patterns cassés dans links/common/src/lib.rs :
         a) derive_key() : zeroize une copie du hash, pas l'original
         b) encrypt_config() : let mut key_copy = *key; key_copy.zeroize()
            → zeroize une copie sur la pile, pas la clé passée en argument
         c) decrypt_config() : idem
         FIX : Utiliser Zeroizing<> wrapper de la crate zeroize autour des
         buffers sensibles, ou zeroizer les GenericArray directement.
         Fichiers : links/common/src/lib.rs
         Difficulté : faible (~30 min)

[✅] 1.0.4  BUG : NaiveDateTime::parse_from_str AVEC FORMATS DATE-ONLY
         handle_killdate_command() utilise NaiveDateTime::parse_from_str
         avec les formats "%Y-%m-%d" et "%Y/%m/%d" qui ne contiennent
         pas de composante temporelle → échouent systématiquement.
         Conséquence : "killdate 2026-12-31" ne marche pas (seul le
         timestamp Unix fonctionne).
         FIX : Utiliser NaiveDate::parse_from_str pour les formats
         date-only, puis .and_hms_opt(23, 59, 59) pour convertir.
         Fichiers : links/common/src/lib.rs (handle_killdate_command)
         Difficulté : faible (~15 min)

 SPRINT 1.5 — DETTE TECHNIQUE DÉCOUVERTE PAR AUDIT (P0, ~3 jours)
================================================================================
Problèmes de qualité de code qui impactent la maintenabilité et la fiabilité.

[🔴] 1.5.1  FACTORISER link_loop() DANS LINK-COMMON
         La boucle C2 (stage 1/2/3) est dupliquée ~150 LOC × 3 implants
         (linux, osx, windows). Les seules différences sont :
         - username(), hostname(), platform_info() (spécifiques à l'OS)
         - dispatch() (commandes OS-spécifiques)
         Viole la règle "code commun → links/common/".
         Solution KISS : trait LinkPlatform avec les méthodes OS-spécifiques,
         link_loop() générique dans link-common qui utilise le trait.
         Alternativement : une seule fonction link_loop() dans common
         qui prend des closures pour les parties OS-spécifiques.
         Fichiers : links/common/src/lib.rs (nouveau module loop.rs),
                    links/*/src/stdlib.rs (simplifier à ~50 LOC chacun)
         Difficulté : moyenne (~2-3h)
         Impact : supprime ~300 LOC de duplication + réduit le risque
         de divergence entre implants.

[🟡] 1.5.2  IMPL DEFAULT POUR TaskResponse
         Le struct TaskResponse dans link-common n'implémente pas Default,
         forçant la construction manuelle avec 8 champs à chaque fallback
         de déchiffrement (6 occurrences dans le code).
         FIX : Ajouter #[derive(Default)] à TaskResponse (tous les champs
         ont déjà des valeurs par défaut via serde).
         Fichiers : links/common/src/lib.rs
         Difficulté : triviale (~5 min)

[🟡] 1.5.3  MUTEX .unwrap() INCOHÉRENT DANS CLI
         Le CLI (cli.rs) utilise .unwrap() sur les locks Mutex alors que
         le serveur (routes.rs) utilise .unwrap_or_else(|e| e.into_inner()).
         Si un thread panic, le CLI crashera.
         FIX : Aligner cli.rs sur le pattern du serveur.
         Fichiers : server/src/cli.rs (~10 occurrences)
         Difficulté : faible (~15 min)

[🟡] 1.5.4  CallbackPayload DÉFINI DANS LA BOUCLE
         Le struct #[derive(Serialize)] CallbackPayload est défini dans
         le corps de la boucle loop {} dans les 3 implants.
         Pas un bug (Rust compile correctement) mais non-idiomatique.
         FIX : Déplacer en scope module. Sera résolu par 1.5.1 si
         link_loop() est factorisé.
         Fichiers : links/*/src/stdlib.rs
         Difficulté : triviale

[🟡] 1.5.5  LinkyError::From<anyhow::Error> PERD L'INFORMATION
         L'impl From<anyhow::Error> dans error.rs convertit tout en
         BuildFailed(String), ce qui est lossy et non-idiomatique.
         FIX : Soit ajouter un variant Generic(String) ou AnyhowError(String),
         soit supprimer cette impl et utiliser anyhow directement là où
         c'est nécessaire.
         Fichiers : server/src/error.rs
         Difficulté : faible

[✅] 1.5.6  TRACING INITIALISÉ AVEC FILTRE "off"
         main.rs initialise le subscriber tracing avec .with_env_filter("off")
         ce qui supprime TOUS les logs (info, warn, error).
         Les tracing::info!() / tracing::error!() dans generate.rs et
         routes.rs sont donc silencieux.
         FIX : Utiliser "warn" par défaut ou "linky=info" et respecter
         la variable d'environnement RUST_LOG.
         Fichiers : server/src/main.rs
         Difficulté : triviale (~5 min)

[✅] 1.5.7  #[allow(dead_code)] OBSOLÈTE SUR Link::secret
         Le champ Link::secret est activement lu dans routes.rs stage3
         (l.secret.clone()). L'annotation #[allow(dead_code)] est obsolète
         et devrait être retirée.
         Link::external_ip est aussi marqué dead_code mais est rempli
         au register — pourrait être affiché dans show_info().
         Fichiers : server/src/links.rs
         Difficulté : triviale

[🟡] 1.5.8  encrypt_payload/decrypt_payload SONT DES ALIAS INUTILES
         Dans link-common, encrypt_payload() et decrypt_payload() sont
         de simples alias vers encrypt_config()/decrypt_config().
         Cela crée de la confusion — on ne sait pas quelle fonction
         utiliser. Le serveur a ses propres versions de encrypt/decrypt_payload.
         FIX : Soit supprimer les alias (utiliser encrypt_config partout),
         soit les différencier réellement (ex: payload ajoute un MAC,
         config non).
         Fichiers : links/common/src/lib.rs
         Difficulté : faible

[🟡] 1.5.9  split_first() DUPLIQUÉ
         La fonction split_first() existe dans link-common ET dans cli.rs.
         Le serveur ne dépend pas de link-common (et ne devrait pas pour
         éviter de tirer les deps implant), mais la duplication est fragile.
         FIX : Soit extraire dans un micro-crate "linky-util", soit
         accepter la duplication avec un commentaire.
         Fichiers : links/common/src/lib.rs, server/src/cli.rs
         Difficulté : faible

      Le paramètre `interface` n'est jamais utilisé dans le filtrage —
      retourne parfois l'IP d'une mauvaise interface.

      les noms recommencent à link-1. Confusion possible en engagement.

[BUG-MINEUR] OSX : commandes "info", "ps", "netstat" non implémentées.

      output brut non structuré. → voir item 4.4

[✅ RÉSOLU] Tracing filter : "off" → "warn" (RUST_LOG respecté). (1.5.6)
