[ ] 2.1  FILE DOWNLOAD (exfiltration: implant → serveur) (P0)

[ ] 2.2  FILE UPLOAD (infiltration: serveur → implant) (P0)
         Ajouter une commande "upload <local_path> <remote_path>" côté CLI.
         Le serveur encode le fichier en base64 et l'envoie comme payload

── Réseau ─────────────────────────────────────────────────────────────────────

[ ] 3.1  MALLEABLE C2 PROFILES (P1)
         Rendre configurable à la génération :
         - User-Agent (actuellement hardcodé IMPLANT_UA dans routes.rs +
           UA dans les implants)
         - Cookie name/value (actuellement "banner=banner")
         - URL paths pour les 3 stages (actuellement /js, /static/register,
           /static/get)
         - Headers HTTP additionnels
         - Body encoding (JSON, form-urlencoded, base64 dans un champ)
         Implémenter via un fichier de profil TOML/YAML lu par generate.rs
         et compilé dans l'implant via build.rs env vars.

[ ] 3.2  SUPPORT REDIRECTEURS (P1)
         Documenter l'utilisation avec Apache/Nginx mod_rewrite.
         Le serveur doit aussi accepter les connexions sans valider l'IP
         source (déjà le cas) et supporter le header X-Forwarded-For pour
         logger la vraie IP de l'implant.

[ ] 3.3  DOMAIN FRONTING (P2)
         Support de l'en-tête Host différent du SNI TLS pour passer par
         des CDN (CloudFront, Azure CDN). L'implant doit pouvoir configurer
         le SNI et le Host header séparément.

[ ] 3.4  CANAUX DE COMMUNICATION ALTERNATIFS (P2)
         - DNS over HTTPS (DoH) comme canal C2 (encode les données dans les
           queries DNS, utilise un résolveur DoH comme transport)
         - WebSocket comme alternative au polling HTTP
         - SMB named pipes pour le mouvement latéral (P2P entre implants)

[ ] 3.5  PROXY-AWARE (Windows) (P1)
         L'implant Windows devrait détecter automatiquement le proxy système
         (WinINet / Registry / WPAD) et s'en servir pour les connexions.
         reqwest supporte les proxys via .proxy(), mais il faut d'abord
         détecter la config proxy du système.

── Windows — anti-EDR ─────────────────────────────────────────────────────────

[ ] 3.6  INDIRECT SYSCALLS (P1)
         L'injection actuelle (links/windows/src/stdlib.rs fn inject_shellcode)
         utilise directement VirtualAllocEx → WriteProcessMemory →
         VirtualProtectEx → CreateRemoteThread via les crates windows/winapi.
         Ces appels sont hookés par tous les EDR modernes.
         Implémenter des syscalls indirects (Hell's Gate / Halo's Gate /
         SysWhispers3) pour bypasser les hooks userland ntdll.

[ ] 3.7  AMSI BYPASS (P1)
         Patcher amsi.dll!AmsiScanBuffer en mémoire avant toute exécution
         PowerShell. Nécessaire pour que la commande "powershell" du CLI
         fonctionne sans déclencher AMSI.

[ ] 3.8  ETW BYPASS (P1)
         Patcher ntdll!EtwEventWrite pour empêcher la télémétrie ETW de
         reporter les activités de l'implant.

[ ] 3.9  NTDLL UNHOOKING (P1)
         Remapper ntdll.dll depuis le disque (ou depuis KnownDlls) pour
         supprimer tous les hooks EDR en une fois. Techniques :
         - "Fresh copy" : lire ntdll.dll depuis disque, remapper la section
           .text par-dessus celle en mémoire
         - "Perun's Fart" : suspendre les threads, unhook, reprendre

[ ] 3.10 TECHNIQUES D'INJECTION ALTERNATIVES (P2)
         En plus de CreateRemoteThread (actuel), implémenter :
         - Process Hollowing (RunPE)
         - Early Bird APC injection
         - Module Stomping
         - Threadless injection (via callback table manipulation)
         Ajouter des commandes CLI correspondantes.

[ ] 3.11 PPID SPOOFING (P2)
         Utiliser PROC_THREAD_ATTRIBUTE_PARENT_PROCESS pour spoofer le
         parent process lors de la création de processus, masquant la
         hiérarchie de processus suspecte.

[ ] 3.12 DLL / SHELLCODE OUTPUT (P1)
         Actuellement l'implant est généré uniquement en .exe (Windows) ou
         ELF (Linux). Ajouter la génération en DLL (pour DLL sideloading)
         et en shellcode position-independent (pour injection sRDI).
         Fichiers: server/src/generate.rs

── Linux — évasion ────────────────────────────────────────────────────────────

[ ] 3.13 MEMFD_CREATE — EXÉCUTION FILELESS (P1)
         Permettre à l'implant d'exécuter des binaires reçus du serveur
         directement en mémoire via memfd_create() + fexecve() sans
         toucher le disque.

[ ] 3.14 MASQUAGE DU NOM DE PROCESSUS (P2)
         Modifier argv[0] et/ou /proc/self/comm pour masquer le nom du
         processus implant dans la liste des processus.

[ ] 3.15 PTRACE INJECTION (P2)
         Injection de code dans un processus existant via ptrace sur Linux.

[ ] 3.16 LD_PRELOAD PERSISTENCE (P2)
         Utiliser LD_PRELOAD pour injecter l'implant dans des processus
         légitimes au démarrage.

[ ] 7.3  SUPPORT ARM / AARCH64 (P2)

[ ] 7.4  NO_STD PARTIEL (P3)

[ ] 7.5  DAISY CHAINING / P2P (P2)
         Communication implant-à-implant via :
         - SMB named pipes (Windows)
