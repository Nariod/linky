PHILOSOPHIE
───────────
Ce roadmap suit deux principes :
  • KISS — chaque feature doit être la solution la plus simple qui fonctionne.
  • MVP  — livrer un C2 utilisable en engagement réel, pas un clone de Mythic.

Linky ne vise PAS la parité avec Sliver/Mythic/Havoc. L'avantage compétitif est :
  1. Rust natif (pas de Go/Python runtime, RE plus difficile)
  2. Codebase lisible et auditable (~3k LOC vs ~50k pour Sliver)
  3. Déploiement conteneurisé en 3 commandes

Priorités : P0 = bloquant sécurité, P1 = MVP, P2 = nice-to-have, P3 = futur.

 SPRINT 1.5 — DETTE TECHNIQUE (P0 qualité, ~3 jours)

[🔴] 1.5.1  FACTORISER link_loop() DANS LINK-COMMON
         ~150 LOC × 3 implants = ~450 LOC dupliquées. Le seul delta entre
         linux/windows/osx est la fonction dispatch() et les helpers système.
         La boucle C2 complète (3 stages + polling + chiffrement) est
         copiée-collée verbatim.

         Plan de factorisation (KISS) :
         Créer dans link-common/src/loop.rs une fonction publique :

           pub fn run_c2_loop<F>(
               callback: &str,       // env!("CALLBACK") depuis chaque implant
               implant_secret: &str, // env!("IMPLANT_SECRET")
               reg: RegisterRequest,
               dispatch: F,
           ) where F: Fn(&str) -> String

         Chaque stdlib.rs devient alors ~40 LOC :
           - Fonctions système (username, hostname, local_ip, platform_info)
           - Fonction dispatch() locale (commandes plateforme-spécifiques)
           - link_loop() qui construit RegisterRequest et appelle run_c2_loop()

         Fichiers : links/common/src/ (nouveau loop.rs ou ajout dans lib.rs),
                    links/linux/src/stdlib.rs,
                    links/windows/src/stdlib.rs,
                    links/osx/src/stdlib.rs
         Difficulté : moyenne (~2h)
         Impact : -300 LOC, maintenance radicalement simplifiée

[✅] 1.5.2  IMPL DEFAULT POUR TaskResponse
         Déjà fait — TaskResponse a #[derive(Default)] dans lib.rs.

[🟡] 1.5.3  MUTEX .unwrap() DANS CLI (server/src/cli.rs)
         ~12 occurrences de links.lock().unwrap() sans poison recovery.
         Remplacer par .unwrap_or_else(|e| e.into_inner()) pour éviter
         un panic si un thread panique en tenant le lock.
         Fichiers : server/src/cli.rs
         Difficulté : triviale (~10 min, sed-friendly)

[✅] 1.5.4  CallbackPayload DÉFINI DANS LA BOUCLE
         Résolu dans 1.5.1 (le struct sera dans link-common/src/loop.rs).
         En attendant : bug mineur, pas de régression fonctionnelle.

[🟡] 1.5.5  LinkyError::From<anyhow::Error> PERD L'INFORMATION
         BuildFailed(err.to_string()) perd la chaîne de causes. Mineur.

[✅] 1.5.6  TRACING INITIALISÉ AVEC FILTRE "off" → "warn"

[🟡] 1.5.7  COMMENTAIRE TROMPEUR DANS stage3_handler (routes.rs)
         Le commentaire du handler dit "at most 2 Mutex acquisitions"
         mais il y en a bien 3 : Lock 1 (resolve), Lock 2 (output + complete),
         Lock 3 (checkin + next task). Corriger le commentaire.
         Fichiers : server/src/routes.rs
         Difficulté : triviale (1 ligne)

[🟡] 1.5.8  encrypt_payload/decrypt_payload ALIAS INUTILES
         Dans link-common/src/lib.rs, ces deux fonctions sont de purs
         alias vers encrypt_config/decrypt_config :
           pub fn encrypt_payload(data: &str, key: &[u8; 32]) -> String {
               encrypt_config(data, key)
           }
         Supprimer les alias, appeler directement encrypt_config/decrypt_config
         aux 3 points d'appel (linux/windows/osx stdlib.rs).
         Fichiers : links/common/src/lib.rs + 3 stdlib.rs
         Difficulté : triviale (~5 min)
         Note : sera résolu naturellement par 1.5.1.

[🟡] 1.5.9  split_first() DUPLIQUÉ
         Présent dans server/src/cli.rs (privé) et links/common/src/lib.rs (pub).
         Le serveur ne dépend pas de link-common → duplication structurelle.
         Action : ajouter un commentaire dans les deux indiquant qu'elles
         doivent rester identiques. Pas de fix architectural possible sans
         créer une dépendance inappropriée.
         Difficulté : triviale (commentaire seulement)

[🟡] 1.5.10 SIGNATURE derive_key INCOHÉRENTE
         routes.rs:    fn derive_key(secret: &[u8], salt: &str)
         link-common:  pub fn derive_key(secret: &str, salt: &str)
         Même algorithme, signatures divergentes = fragile à l'audit.
         FIX : aligner les deux sur &[u8] (plus général).
         Fichiers : links/common/src/lib.rs + tous les appelants dans stdlib.rs
         Difficulté : faible (~10 min)
         Note : sera résolu dans 1.5.1 lors de la refactorisation.
