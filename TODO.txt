 BUGS ACTIFS — DÉCOUVERTS AUDIT #6
================================================================================

  [BUG-MINOR] Link counter non persisté entre redémarrages.

  [BUG-MEDIUM] [B6-1] TCP6/UDP6 — hex_to_ip() ne gère que IPv4 (4 octets LE).
               Les entrées /proc/net/tcp6 ont 8+8 octets little-endian 32 bits
               par groupe. Le résultat affiché est silencieusement corrompu.
               Fichier : links/linux/src/stdlib.rs → parse_net_connections()
               FIX : détecter longueur hex > 8 et appliquer le décodage IPv6.

  [BUG-MINOR]  [B6-2] show_completed_task_results() (cli.rs) est dead code.
               routes.rs marque task.displayed = true en temps réel avant que
               la CLI ne vérifie. La fonction ne s'exécute jamais en pratique.
               FIX : supprimer ou documenter l'intention (fallback headless ?).

  [BUG-MINOR]  [B6-3] Duplication de code crypto en 3 endroits :
               • links/common/src/lib.rs  → derive_key, encrypt_config, decrypt_config
               • server/src/routes.rs     → derive_key, encrypt_payload, decrypt_payload
               • server/src/generate.rs   → derive_key_sha256, encrypt_aes_gcm
               Risque de divergence silencieuse entre server et implant.
               FIX : créer une crate interne server/src/crypto.rs (ou un module
               partagé) et l'utiliser dans routes.rs ET generate.rs.

  [BUG-MINOR]  [B6-4] save_download() utilise un chemin relatif "downloads/" —
               dépend du CWD au lancement du serveur, non documenté.
               FIX : utiliser LINKY_OUTPUT_DIR ou un paramètre de config.

  [BUG-MINOR]  [B6-5] run_c2_loop — stage 1 retry loop sans limite ni backoff
               exponentiel. Un serveur injoignable fait boucler indéfiniment
               sans délai variable, générant du bruit réseau.
               FIX : backoff exponentiel (1s→2s→4s→…→max 60s) + max_retries optionnel.

  [BUG-MINOR]  [B6-6] Erreur réseau stage 3 reset les données de la tâche.
               Sur erreur Err(_), prev_output et prev_task_id sont remis à "".
               Si le réseau est instable, la sortie de la tâche précédente peut
               être perdue avant d'avoir été transmise.
               FIX : conserver prev_output/prev_task_id sur erreur réseau et
               ne les remettre à "" que sur succès (task.q.is_empty()).

  [BUG-MINOR]  [B6-7] parse_upload_args — guillemets simples non supportés,
               et chemins Windows (backslash) peuvent poser problème.
               FIX : supporter 'quotes' et normaliser les séparateurs.

  [BUG-MINOR]  [B6-8] stage3_handler — aucune limite de taille JSON sur /static/get
               (le JsonConfig 64 KB s'applique à tous les handlers, mais un payload
               base64 de fichier upload peut légitimement dépasser 64 KB).
               FIX : augmenter la limite JSON pour /static/get uniquement (ex: 16 MB),
               ou chunker les transferts de fichiers volumineux.

  [BUG-MINOR]  [B6-9] inject_shellcode utilise std::mem::transmute pour caster
               un *mut c_void en fn pointer. C'est UB potentiel selon les règles
               strictes du compilateur Rust pour les raw fn pointers.
               FIX : utiliser une union ou std::mem::transmute_copy avec type explicite
               et bloc unsafe documenté SAFETY:.

 PROPOSITIONS GUI — ANALYSE COMPARATIVE
================================================================================

CONTEXTE
────────
Le projet dispose actuellement d'une CLI rustyline synchrone, mono-thread (main).
L'objectif GUI est d'améliorer l'ergonomie opérateur sans sacrifier le modèle
de sécurité (pas de browser requis, opérateur seul ou petit groupe).

OPTION A — TUI ratatui (★ RECOMMANDÉE)
──────────────────────────────────────
Remplacer rustyline par une interface terminal complète avec layout multi-panneaux.

  Layout proposé :
  ┌─────────────────────┬────────────────────────────────────────┐
  │  LINKS (liste)      │  OUTPUT (scrollable, couleurs)         │
  │  ● link-1|lnx  Actv │  ╔══ link-1 · whoami · 14:38 ═══════╗ │
  │  ○ link-2|win  Inac │  ║ root@hostname                     ║ │
  │                     │  ╚══════════════════════════════════╝ │
  │                     │                                        │
  ├─────────────────────┴────────────────────────────────────────┤
  │ link-1|lnx>  _                                               │
  └──────────────────────────────────────────────────────────────┘

  Avantages :
  • Pas de browser, pas d'exposition réseau supplémentaire
  • Pur Rust, dans l'écosystème existant
  • Idéal pour le modèle opérateur unique
  • Historique des outputs persistent dans le panneau
  • Notifications visuelles sur nouveau link/output (titre de panneau clignotant)
  • Très bon exemple : lazygit, bottom, gitui

  Dépendances à ajouter : ratatui = "0.29", crossterm = "0.28"
  Complexité : ~2 semaines (refactor complet de cli.rs)
  Risque : impact fort sur l'architecture du thread CLI principal

OPTION B — WEB UI EMBARQUÉE (actix-web + HTMX + SSE)
─────────────────────────────────────────────────────
Ajouter un handler web sur un port dédié (ex: 9443 / séparé du C2 8443).
Frontend minimaliste en HTMX, sans framework JS.

  Architecture :
  • Nouveau router actix : GET /ui/* (dashboard), GET /events (SSE stream)
  • Server-Sent Events pour pusher les outputs en temps réel
  • Auth : token Bearer (généré au démarrage, affiché en console)
  • HTTPS self-signed (réutilise rcgen)
  • HTML/CSS intégré comme include_str! (pas de assets externes)

  Avantages :
  • Multi-opérateur (plusieurs browsers connectés)
  • Remote (VPN + browser)
  • Facile à étendre (React plus tard si besoin)
  • Logs persistants dans le browser (historique scroll)

  Inconvénients :
  • Surface d'attaque supplémentaire (second port TLS)
  • Auth à implémenter sérieusement (tokens, CSRF)
  • Complexité accrue du binaire serveur
  • HTMX peut devenir limité pour des interactions riches

  Dépendances : actix-web (déjà présent), tokio::sync::broadcast (déjà présent)
  Complexité : ~3 semaines
  Risque : sécurité (second endpoint), cohérence état server/browser

OPTION C — TAURI (desktop natif Rust + webview)
────────────────────────────────────────────────
Wrapper Tauri autour du serveur Linky, frontend React/Vue dans webview natif.

  Avantages :
  • Expérience desktop native (fenêtre, notifications système)
  • Isolation sécurité Tauri (permissions explicites)
  • Frontend web standard (React, TypeScript)

  Inconvénients :
  • Deux langages (Rust + JS/TS), build chain complexe (Node.js requis)
  • Taille binaire significativement augmentée
  • Containers/headless server plus difficiles
  • Casse le modèle Podman actuel
  • Overkill pour MVP

  Complexité : ~6 semaines
  Risque : élevé (rupture architecture existante)

OPTION D — API REST + FRONTEND DÉCOUPLÉ
─────────────────────────────────────────
Ajouter une API REST complète au serveur, frontend séparé (React, Vue, Svelte).

  Avantages :
  • Séparation maximale des responsabilités
  • Frontend entièrement remplaçable
  • Standard industrie (Swagger/OpenAPI possible)

  Inconvénients :
  • Deux projets à maintenir (server + frontend)
  • Auth et CORS à gérer sérieusement
  • Le plus long à implémenter
  • Deux repos ou monorepo à organiser

  Complexité : ~8 semaines (+ frontend)
  Risque : très élevé pour un MVP mono-opérateur

RECOMMANDATION
──────────────
  Phase immédiate (MVP) : Option A — ratatui TUI
    → Améliore drastiquement l'ergonomie sans risque de sécurité supplémentaire.
    → Reste dans le paradigme "terminal first" cohérent avec le projet.
    → Servira de base pour une future Option B en parallèle.

  Phase suivante (multi-opérateur) : Option B — Web UI HTMX
    → Une fois la TUI stable, ajouter le port web comme option --web-ui.
    → Les deux peuvent coexister (TUI local + Web remote).

  Ne pas implémenter C ou D avant un vrai besoin multi-équipe documenté.
