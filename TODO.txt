PHILOSOPHIE
───────────
Ce roadmap suit deux principes :
  • KISS — chaque feature doit être la solution la plus simple qui fonctionne.
  • MVP  — livrer un C2 utilisable en engagement réel, pas un clone de Mythic.

Linky ne vise PAS la parité avec Sliver/Mythic/Havoc. L'avantage compétitif est :
  1. Rust natif (pas de Go/Python runtime, RE plus difficile)
  2. Codebase lisible et auditable (~3k LOC vs ~50k pour Sliver)
  3. Déploiement conteneurisé en 3 commandes

Priorités : P0 = bloquant sécurité, P1 = MVP, P2 = nice-to-have, P3 = futur.

ÉTAT DE L'ART C2 2026 — OÙ SE SITUE LINKY
───────────────────────────────────────────
  Critère              Linky   Sliver  Mythic  Havoc
  ─────────────────────────────────────────────────
  Transport crypto      2/10   9/10    9/10    8/10
  Payload crypto        1/10   9/10    9/10    8/10
  Évasion EDR           0/10   8/10    7/10    9/10

  GUI/UX                2/10   7/10    10/10   9/10
  CI/Tests              2/10   9/10    8/10    6/10

Score global Linky : ~18% vs 85-95% pour les références.
Objectif MVP : atteindre ~50% — suffisant pour des engagements simples.

 SPRINT 0.5 — DETTE TECHNIQUE DÉCOUVERTE (P0, ~3-4 jours)
================================================================================
Issues découvertes lors de la revue du 2026-03-25. Doivent être résolues
AVANT le Sprint 1 car elles affectent la stabilité et la maintenabilité.

── Factorisation du code ──────────────────────────────────────────────────────

[ ] 0.5.1 MIGRER l'implant OSX vers link-common
          Problème : links/osx/src/stdlib.rs duplique RegisterRequest,
          CallbackRequest, TaskResponse, et build_client() localement.
          L'implant OSX ne dépend même pas de link-common (voir Cargo.toml).
          Solution :
          - Ajouter link-common comme dépendance dans links/osx/Cargo.toml
          - Supprimer les structs/fonctions dupliquées dans stdlib.rs
          - Utiliser les imports de link-common
          - Ajouter les deps manquantes : rand, chrono, aes-gcm, sha2, hex
          Fichiers : links/osx/Cargo.toml, links/osx/src/stdlib.rs

[ ] 0.5.2 FACTORISER les fonctions partagées Linux/Windows → link-common
          Les fonctions suivantes sont identiques ou quasi-identiques :
          - sleep_with_jitter()          → link_common::sleep_with_jitter()
          - handle_sleep_command()       → link_common::handle_sleep_command()
          - handle_killdate_command()    → link_common::handle_killdate_command()
          - split_first()               → link_common::split_first()
          - download_file()             → link_common::download_file()
          - upload_file()               → link_common::upload_file()
          - should_exit()               → link_common::should_exit()
          - Atomics SLEEP/JITTER/KILL   → link_common (module state.rs)
          Le module common devra exposer les setters/getters pour les atomics.
          Fichiers : links/common/src/lib.rs (ou sous-modules),
                     links/linux/src/stdlib.rs, links/windows/src/stdlib.rs,
                     links/osx/src/stdlib.rs

[ ] 0.5.3 EXTRAIRE la logique de dispatch commune
          Les commandes cd, pwd, ls, pid, whoami, shell, sleep, killdate,
          download, upload sont les mêmes sur Linux/OSX.
          Solution KISS : un fn dispatch_common() dans link-common qui
          retourne Option<String>. Si None, l'implant platform-specific
          gère la commande.
          Fichiers : links/common/src/dispatch.rs (nouveau)

── Bugs fonctionnels ──────────────────────────────────────────────────────────

[ ] 0.5.4 CORRIGER la fenêtre console Windows sur tasklist/netstat
          list_processes() et list_network_connections() dans
          links/windows/src/stdlib.rs lancent Command::new("tasklist")
          et Command::new("netstat") SANS le flag CREATE_NO_WINDOW.
          → Visible sur le desktop de la victime = détection immédiate.
          Solution : Ajouter le trait CommandExt + creation_flags(0x0800_0000)
          comme c'est déjà fait dans shell_exec().
          Pattern KISS : extraire un helper fn silent_command(prog, args)
          qui applique le flag automatiquement.
          Fichier : links/windows/src/stdlib.rs

[ ] 0.5.5 CORRIGER le conflit ps/powershell dans le CLI
          Dans cli.rs, interact(), le match "ps" est mappé sur powershell :
            "powershell" | "ps" => { ... format!("powershell ...") }
          Mais l'implant comprend "ps" comme process listing.
          Solution : Supprimer "ps" comme alias de powershell.
          Garder uniquement "powershell" pour PowerShell.
          Ajouter "ps" comme commande qui envoie "ps" à l'implant.
          Fichier : server/src/cli.rs

[ ] 0.5.6 ÉCRIRE les fichiers téléchargés sur disque (serveur)
          download_file() côté implant renvoie "FILE:<path>:<base64>".
          routes.rs parse correctement et stocke dans task.file_content,
          mais le fichier n'est JAMAIS écrit sur le filesystem du serveur.
          L'opérateur n'a aucun moyen de récupérer le fichier.
          Solution :
          - Créer un répertoire downloads/<link_name>/ dans output_dir()
          - Quand parse_file_response() réussit, décoder le base64 et
            écrire dans downloads/<link_name>/<filename>
          - Afficher le chemin local dans le message UI
          Fichiers : server/src/routes.rs, éventuellement server/src/files.rs

── Robustesse serveur ─────────────────────────────────────────────────────────

[ ] 0.5.7 SIMPLIFIER le lock management dans stage3_handler
          stage3_handler() dans routes.rs acquiert/relâche le Mutex
          4 à 5 fois dans un seul appel HTTP. C'est fragile et
          difficile à auditer.
          Solution : Extraire la logique dans Links::process_callback()
          qui prend le body + x_request_id et retourne un TaskResponse.
          Un seul lock dans le handler.
          Alternative KISS : au minimum, regrouper les opérations
          pour réduire à 2 acquisitions max (submit output + get next).
          Fichier : server/src/routes.rs, server/src/links.rs

[ ] 0.5.8 VALIDER les inputs RegisterRequest
          Aucune validation des champs reçus à l'enregistrement.
          Un implant malveillant ou un fuzzer peut envoyer des strings
          de plusieurs Mo → OOM sur le serveur.
          Solution KISS : tronquer chaque champ à une taille raisonnable
          (username: 256, hostname: 256, ip: 45, platform: 64).
          Fichier : server/src/routes.rs (stage2_handler)

[ ] 0.5.9 CONFIGURER actix-web payload limits
          Par défaut, actix-web accepte des body JSON jusqu'à 256 KB.
          C'est probablement suffisant mais doit être explicite.
          Ajouter web::JsonConfig::default().limit(65536) dans App::new().
          Fichier : server/src/server.rs

 SPRINT 1 — SÉCURITÉ FONDAMENTALE (P0, ~2 semaines)
================================================================================
Sans ce sprint, Linky n'est pas utilisable en engagement.

── Cryptographie ──────────────────────────────────────────────────────────────

[ ] 1.1  CLÉ UNIQUE PAR IMPLANT À LA GÉNÉRATION
         Problème : derive_key("linky-secret-key", "callback-salt") est
         identique pour TOUS les implants. Un seul reverse → toutes les
         configs compromises.
         Solution KISS :
         - generate.rs génère un secret aléatoire 32 bytes (hex) par build
         - Injecté via build.rs comme env var CARGO : IMPLANT_SECRET
         - Le serveur stocke le secret dans la struct Link au register
         - L'implant dérive sa clé avec derive_key(IMPLANT_SECRET, salt)
         Note : nécessite que 0.5.2 soit fait pour centraliser derive_key.
         Fichiers : server/src/generate.rs, links/*/build.rs,
                    links/common/src/lib.rs

[ ] 1.2  CHIFFREMENT DES PAYLOADS C2 (stage 3)
         Problème : les commandes et résultats transitent en JSON clair.
         Solution KISS (sans ECDH pour le MVP) :
         - Le secret par implant (1.1) sert de clé de session
         - Chaque body JSON est chiffré AES-256-GCM avant envoi
         - Le nonce est préfixé au ciphertext (12 bytes)
         - Header Content-Type reste application/json, le body est
           { "data": "<hex_encrypted>" }
         Fichiers : links/common/src/lib.rs (encrypt_payload/decrypt_payload)
                    server/src/routes.rs, links/*/src/stdlib.rs

[ ] 1.3  OBFUSCATION DES STRINGS STATIQUES (MVP)
         Problème : UA, cookie "banner=banner", routes "/js", "/static/*"
         sont des signatures triviales pour IDS/EDR.
         Solution KISS (pas de litcrypt, juste du XOR compiletime) :
         - Macro simple qui XOR les strings au compile-time
         - Déchiffrement au runtime via un const fn
         - OU utiliser le crate obfstr (zero-dep, mature)
         Fichiers : links/*/src/stdlib.rs

── Nettoyage sécurité ─────────────────────────────────────────────────────────

[ ] 1.4  ZEROIZE LES CLÉS EN MÉMOIRE
         Ajouter zeroize::Zeroize sur les buffers de clé après usage.
         Le crate zeroize est déjà une dep transitive de rustls.
         Fichiers : links/common/src/lib.rs

── Profils malléables (version KISS) ──────────────────────────────────────────

[ ] 2.1  PROFILS C2 CONFIGURABLES À LA GÉNÉRATION
         Pas de fichier TOML/YAML complexe — juste des env vars cargo :
         - CALLBACK (existe déjà)
         - C2_UA (User-Agent, défaut = Chrome récent)
         - C2_COOKIE_NAME / C2_COOKIE_VALUE (défaut = aléatoire)
         - C2_ROUTE_STAGE1, C2_ROUTE_STAGE2, C2_ROUTE_STAGE3
         Injectés via build.rs, utilisés comme const dans l'implant.
         Le serveur doit aussi lire ces valeurs (config file ou env).
         Prérequis : 0.5.2 (factorisation common)
         Fichiers : links/*/build.rs, server/src/routes.rs,
                    server/src/generate.rs

── Taille binaire ─────────────────────────────────────────────────────────────

[ ] 2.2  OPTIMISER LA TAILLE DES IMPLANTS
         Ajouter dans les Cargo.toml des implants :
         [profile.release]
         opt-level = "z"
         lto = true
         codegen-units = 1
         panic = "abort"
         strip = true
         Objectif : < 2 MB pour l'implant Linux (actuellement ~8 MB).
         Note : peut se faire dans le workspace Cargo.toml racine avec
         des profils par package.

── Évasion Windows (le strict minimum) ────────────────────────────────────────

[ ] 2.3  INDIRECT SYSCALLS POUR L'INJECTION
         Remplacer VirtualAllocEx/WriteProcessMemory/CreateRemoteThread
         par des syscalls indirects (Hell's Gate pattern).
         Crate possible : rust-syscalls ou implémentation manuelle.
         Scope KISS : seulement les 4 syscalls d'injection, pas tout ntdll.
         Fichier : links/windows/src/stdlib.rs (fn inject_shellcode)

[ ] 2.4  AMSI PATCH (une seule technique, la plus simple)
         Patcher AmsiScanBuffer pour retourner AMSI_RESULT_CLEAN.
         Nécessaire pour que "powershell" fonctionne sans alerte.
         Fichier : links/windows/src/stdlib.rs (nouveau fn patch_amsi)

[ ] 2.5  ETW PATCH
         Patcher EtwEventWrite pour supprimer la télémétrie.
         Pattern identique à AMSI : écrire un ret au début de la fonction.
         Fichier : links/windows/src/stdlib.rs (nouveau fn patch_etw)

 BUGS CONNUS

[BUG] File upload: les implants Linux/Windows ignorent task.upload et
      task.upload_path dans link_loop() — la commande est envoyée comme
      "upload <remote_path>" sans le contenu base64.
      FIX : lire task.upload/task.upload_path et les passer à dispatch().
      STATUS : PARTIELLEMENT CORRIGÉ — le code de link_loop() construit
      maintenant la commande "upload <content> <path>" mais le parsing
      reste fragile (split sur le premier espace).

[BUG] Linux info: get_interface_ip() utilise /proc/net/fib_trie mais
      la logique de parsing est fragile — retourne parfois l'IP d'une
      mauvaise interface. Le paramètre `interface` n'est d'ailleurs
      jamais utilisé dans la logique de filtrage.

[BUG] File download côté serveur : le contenu base64 est stocké dans
      task.file_content mais n'est jamais écrit sur disque. L'opérateur
      ne peut pas récupérer les fichiers téléchargés.
      → Voir item 0.5.6

[BUG] Windows: list_processes() et list_network_connections() n'utilisent
      pas CREATE_NO_WINDOW — fenêtre console visible sur le desktop.
      → Voir item 0.5.4

[BUG] CLI: "ps" est un alias de powershell au lieu de process listing.
      → Voir item 0.5.5

[BUG-MINEUR] OSX: implant sans encryption, jitter, file ops, ou kill date.
      → Sera résolu par 0.5.1 + 0.5.2 + 4.4

[BUG-MINEUR] links counter dans Links::add_link() n'est pas persisté.
      Après restart du serveur, les noms recommencent à link-1.
      Pas bloquant pour le MVP, mais risque de confusion en engagement.

 ORDRE D'EXÉCUTION RECOMMANDÉ

  1. Sprint 0.5.1 (OSX → link-common)     ← débloque la factorisation
  2. Sprint 0.5.2 (factoriser Linux/Win)   ← réduit ~400 LOC de duplication
  3. Sprint 0.5.3 (dispatch commun)        ← optionnel mais réduit encore

  5. Sprint 1.1 (clé par implant)          ← fondation crypto
  6. Sprint 1.2 (chiffrement payload)      ← dépend de 1.1
  7. Sprint 1.3 (obfuscation strings)      ← indépendant
  8. Sprint 1.4 (zeroize)                  ← rapide, indépendant
  9. Sprint 2.2 (taille binaire)           ← rapide, gros impact
  10. Sprint 4.1-4.3 (tests + CI)          ← valider tout ce qui précède
  11. Sprint 2.1 (profils malléables)      ← dépend de 0.5.2
  12. Sprint 2.3-2.5 (évasion Windows)     ← complexe, après stabilisation
  13. Sprint 3.x (post-exploitation)       ← features additionnelles

Estimation totale MVP : ~6-8 semaines de travail effectif.
